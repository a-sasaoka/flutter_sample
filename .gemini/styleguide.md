# Style Guide (Ver 2.0)

## 1. PRの要約（サマリー）に対する指示

必ず以下のルールに従ってPRの説明を生成してください。

- 説明、要約、変更点のリストなど、すべてのテキストは「日本語」で出力してください。
- プログラムの変数名などはそのまま英語で構いませんが、解説文は日本語に徹してください。
- 英語での出力は一切禁止します。
- Respond strictly in Japanese.

## 2. コードレビューに対する指示

必ず以下のルールと観点に従ってコードレビューを行ってください。

### 【基本ルール】

- レビューコメントや解説は、すべて「日本語」で出力してください。
- 英語での出力は一切禁止します。
- Respond strictly in Japanese.
- 修正案のコードを提示する場合は、具体的な理由も添えてください。
- **Dart 3 構文（Records, Patterns, Sealed classes）の積極的な活用を推奨してください。**

### 【レビューの観点】

#### 1. アーキテクチャと依存性注入 (DI)

- **コンストラクタ注入の徹底**: Data層やDomain層のクラスが `Ref` に直接依存していないか？
  - 💡 **理由**: 特定のフレームワークへの結合を避け、純粋な Dart クラスとして高速かつ安定した単体テストを可能にするため。
- **Feature-Driven Architecture**: 適切なフォルダ構成（features/機能名/data, domain, application, presentation）になっているか？
- **ロギング**: `print()` や `logger` パッケージが残っていないか？（すべて `talker_flutter` を使用しているか）

#### 2. Riverpod 状態管理と Dart 3 構文

- **Record 型の分解 (Destructuring)**: 複数の Provider を合体させた Record データを取得する際、分解構文を使っているか？
  - ❌ `final theme = config.theme;`
  - ✅ `final (:theme, :locale) = config;`
- **パターンマッチング (if-case)**: `AsyncValue` のエラー処理やキャッシュの型チェックで `if-case` を使っているか？
  - 💡 **理由**: 強制アンラップ (`!`) や手動キャスト (`as`) を排除し、型安全に値を抽出するため。
- **購読方法**: ウィジェットの build メソッド内で `ref.read` を使わず、必ず `ref.watch` を使用しているか？
- **イミュータブルな更新**: コレクション（List等）を更新する際、破壊的な変更（.add等）を避け、`state = [...state]` のように新しいインスタンスを代入しているか？

#### 3. UI 設計とアクセシビリティ

- **Material 3 準拠**: `colorScheme` を適切に使用しているか？
  - 💡 **ポイント**: 補助的なテキストには `onSurfaceVariant` を使い、視覚的な階層（コントラスト）を最適化する。
- **非同期処理の安全性**: 非同期操作（`Future`）の後の画面遷移などで `context.mounted` チェックを行っているか？

#### 4. テストコード（最重要）

- **多言語化 (L10n) のモック**: `AppLocalizations` をモック化しているか？
  - 💡 **理由**: 翻訳テキストの変更が UI テストの成否に影響しないようにし、保守性を高めるため。
- **UI テストの安定化**:
  - `tester.view.physicalSize` を固定（例: 1080x1920）しているか？（実行環境によるタップ失敗の防止）
  - スクロールを伴うテスト（Pull-to-Refresh等）で `AlwaysScrollableScrollPhysics` を考慮しているか？
- **AutoDispose 対策**: テスト内で `container.listen` を使い、プロバイダの予期せぬ破棄を防いでいるか？
- **非同期エラーテスト**: エラー系のテストで `.future` を待機してデッドロック（タイムアウト）を起こす書き方をしていないか？

#### 5. 初心者向けの解説

- 解説のステップ: 指摘箇所について、以下の3ステップで解説してください：
  1. **現状の問題点**: 何が起きているか、または改善の余地があるか。
  2. **なぜそれが良くないのか**: 保守性、テスト容易性、安全性の観点からの理由。
  3. **修正案**: 具体的なコード例とそのメリット。
